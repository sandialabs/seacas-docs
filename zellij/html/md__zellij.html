Zellij is a "mesh concatenation" application for generating a mesh consisting of a "lattice" containing one or more "unit cell" template meshes. The lattice is a two-dimensional arrangement of the unit cell template meshes.

The unit cell template meshes are placed by zellij into the specified locations in the lattice and the nodes on the boundaries of the unit cell meshes are united or coincident. Each unit cell mesh must have the same exterior boundary meshes and coordinate extents on the X and Y coordinate faces, but the Z faces are only required to have the same coordinate extent; the Z face meshes are not required to be the same among the different unit cells.

The lattice can be represented as a IxJ regular grid with each "cell" in the grid or lattice containing one of the unit cell template meshes.

The only required option is -lattice followed by the name of the file containing the lattice description. The other options are used to specify compression of the output file; the format of the output file; or to request additional debug output.

If the -output <filename> option is not specified, then the output mesh will be named zellij-out.e.

Lattice Description File Format

The format of the lattice description file is fairly simple, but is also very rigid. There are two sections of the file – the unit cell dictionary and the lattice definition.

Unit Cell Dictionary

The unit cell dictionary defines the unit cell template meshes that will be placed in the lattice. The dictionary begins with a line containing BEGIN_DICTIONARY followed by one or more lines defining the unit cells and is then ended with a line containing END_DICTIONARY

The syntax of the lines defining the unit cells consists of two fields – an arbitrary key and the filename containing the Exodus file defining the mesh for this unit cell. The only restriction on the key is that it must be unique in the dictionary. The filenames must specify the path (either absolute or relative to the current execution directory) to the Exodus file; it can optionally be delimited by double quotes ("</tt>). The filenames do not need to be unique, but it is more efficient in both memory and time if each unit cell template mesh is unique. As an example, here is a valid dictionary definition: @icode BEGIN_DICTIONARY 0001 "../zellij-example/xatom-1b.e" 0002 "../zellij-example/xatom-Y.e" 0003 "../zellij-example/xatom-X.e" 0004 "../zellij-example/xatom-2b.e" END_DICTIONARY @endicode The unit cell dictionary must appear before the lattice definition in the lattice description file. If an error is detected during the parsing of the unit cell dictionary, the code will output an error message and terminate. Errors can be incorrect syntax, missing unit cell template meshes, duplicate keys, or problems reading the mesh description from a unit cell template mesh. The unit cell template mesh file is accessed and partially read at the time that zellij parses the corresponding unit cell dictionary line. @subsection autotoc_md4 Lattice Definition The lattice definition specifies the size of the lattice and the distribution of the unit cell(s) within that lattice. The lattice definition must follow the unit cell dictionary in the lattice description file. The first line of the lattice definition begins with the line <tt>BEGIN_LATTICE {i} {j} 1</tt> where <tt>{i}</tt> and <tt>{j}</tt> specify the size of the <tt>IxJ</tt> arrangement of unit cells. For example, the line <tt>BEGIN_LATTICE 5 5 1</tt> would define a lattice containing 25 unit cell instances arranged in a 5 by 5 regular grid. The last line of the lattice definition is the line <tt>END_LATTICE</tt>. When that line is encountered, zellij will begin outputting the mesh. Between the <tt>BEGIN_LATTICE</tt> and <tt>END_LATTICE</tt> are <tt>{j}</tt> lines with <tt>{i}</tt> entries per line. The entries are any of the _key_s that were specified in the unit cell dictionary. As an example, here is a valid lattice definition using the keys of the example dictionary from the previous section: @icode BEGIN_LATTICE 5 5 1 0001 0002 0003 0002 0001 0002 0003 0003 0003 0002 0003 0003 0004 0003 0003 0002 0003 0003 0003 0002 0001 0002 0003 0002 0001 END_LATTICE @endicode Although the lattice is typically symmetric and square, this is not a requirement and is not checked. If an error is detected during the parsing of the <em>lattice</em>, the code will output an error message and terminate. Errors can include invalid keys, incorrect number of lattice definition lines, or incorrect number of keys on a definition line. Note that zellij does not require that the unit cell keys be numeric; the following example shows a different method for specifying the same lattice definition file as the previous example: @icode BEGIN_DICTIONARY - "../zellij-example/xatom-1b.e" | "../zellij-example/xatom-Y.e" + "../zellij-example/xatom-X.e" * "../zellij-example/xatom-2b.e" END_DICTIONARY BEGIN_LATTICE 5 5 1 - | + | - | + + + | + + * + + | + + + | - | + | - END_LATTICE @endicode @section autotoc_md5 Unit Cell Template Mesh Requirements Zellij requires that the boundary mesh (<tt>X</tt> and <tt>Y</tt> faces) of each of the unit cell templates be a <em>regular</em> "structured" mesh. Basically this means that the faces of the mesh elements on the boundary are in a regular rectangular grid such that each mesh face is rectangular (90 degree corners) and that the boundary mesh on the minimum <tt>X</tt> face is the same as that on the maximum <tt>X</tt> face and similarly for the minimum <tt>Y</tt> face and the maximum <tt>Y</tt> face. Additionally, the X faces on <em>all</em> unit cells must match and the Y faces on <em>all</em> unit cells must match both in structure and in coordinate extent. This requirement is verified during execution. The <tt>Z</tt> faces are less constrained with the only requirement being that the coordinate extents of all <tt>Z</tt> faces must be the same (which follows from the <tt>X</tt> and <tt>Y</tt> face requirement); the structure of the mesh on the <tt>Z</tt> faces is arbitrary. The unit cell meshes can contain any number of element blocks; however, each element block <em>must</em> contain hexahedral elements with 8-nodes per element. The element blocks do not need to be the same in each unit cell mesh, but if they do share the same element block <tt>id</tt>, then those elements will be combined into the same element block in the output mesh with the same <tt>id</tt>. The output mesh will contain the union of all element blocks existing on the input mesh unit cells. For example, if: * unit cell <tt>0001</tt> has element blocks <tt>1 10 100</tt> * unit cell <tt>0002</tt> has element blocks <tt>2 20 200</tt> * unit cell <tt>0003</tt> has element blocks <tt>1 2 10 20</tt> * unit cell <tt>0004</tt> has element blocks <tt>10 20 100 200</tt> The output mesh will have element blocks <tt>1 2 10 20 100 200</tt> @section autotoc_md6 Sideset Handling By default, zellij will replicate any sidesets that are defined on the input unit cell meshes to the output mesh file. The sidesets will have the same names and ids as the sidesets on the input unit cell meshes. If you do not want the sidesets replicated, you can add the command line option <tt>-ignore_sidesets</tt> and any sidesets on the input unit cell meshes will be ignored. Zellij can also generate new sidesets on the boundaries of the output mesh via the command line option <tt>-generate_sidesets \<axes\></tt> where <tt>axes</tt> is one or more letters specifying the face of the output mesh on which to generate a sideset. Valid letters are <tt>xyzXYZ</tt> or <tt>ijkIJK</tt> which correspond to: * <tt>x</tt> or <tt>i</tt> for surface on minimum X coordinate (default name = <tt>min_i</tt>) * <tt>y</tt> or <tt>j</tt> for surface on minimum Y coordinate (default name = <tt>min_j</tt>) * <tt>z</tt> or <tt>k</tt> for surface on minimum Z coordinate (default name = <tt>min_k</tt>) * <tt>X</tt> or <tt>I</tt> for surface on maximum X coordinate (default name = <tt>max_i</tt>) * <tt>Y</tt> or <tt>J</tt> for surface on maximum Y coordinate (default name = <tt>max_j</tt>) * <tt>Z</tt> or <tt>K</tt> for surface on maximum Z coordinate (default name = <tt>max_k</tt>) For example <tt>-generate_sidesets xyXY</tt> would generate sideset on the surfaces corresponding to the minimum and maximum X and Y coordinates on the output mesh. By default, the generated sidesets will be named as shown in the list above. The names can be changed with the <tt>-sideset_names \<arg\></tt> command line option. The syntax of <tt>\<arg\></tt> is <tt>axis:name,axis:name,...</tt> where <tt>axis</tt> is one of <tt>ijkIJK</tt> or <tt>xyzXYZ</tt> and <tt>name</tt> is the name of the specified sideset. For example, <tt>-sideset_names x:left,X:right</tt> would name the sidesets on the minimum x and maximum X faces <tt>left</tt> and <tt>right</tt> respectively. There will be an error if two or more sidesets have the same name. @section autotoc_md7 Parallel Execution Zellij can produce a mesh decomposed into a file-per-rank for use in a parallel analysis application. Note that Zellij itself is run serially. The main option that tells Zellij to produce the decomposed files is <tt>-ranks \<number_of_ranks\></tt>. If this is specified, then Zellij will create <tt>number_of_ranks</tt> individual files each containing a portion of the complete model. The files will have the information needed by a parallel application to read the data and set up the correct communication paths and identify the nodes that are shared across processor boundaries. The decomposition method can also be specified. This determines the algorithm that is used to break the lattice into <tt>number_of_ranks</tt> pieces each with approximately the same computational complexity. The decomposition methods are: * <tt>-rcb</tt> Use recursive coordinate bisection method to decompose the input mesh in a parallel run. * <tt>-rib</tt> Use recursive inertial bisection method to decompose the input mesh in a parallel run. * <tt>-hsfc</tt> Use hilbert space-filling curve method to decompose the input mesh in a parallel run. * <tt>-linear</tt> Use the linear method to decompose the input mesh in a parallel run. Elements in order first <tt>n/p</tt> to proc 0, next to proc 1. * <tt>-cyclic</tt> Use the cyclic method to decompose the input mesh in a parallel run. Elements handed out to <tt>id % proc_count</tt>. * <tt>-random</tt> Use the random method to decompose the input mesh in a parallel run. Elements are assigned randomly to processors in a way that preserves balance (do <em>not</em> use for a real run)) The <tt>-hsfc</tt> method is the default if no other decomposition method is specified. Note that the decomposition occurs at the <em>grid</em> level so the elements of each grid cell will not be split across multiple ranks. The grid cells are weighted by the number of elements in the cell which should produce a balanced decomposition if there are unit cells of varying element counts. The <tt>-linear</tt>, <tt>-cyclic</tt>, and <tt>-random</tt> methods are typically used for debugging and testing Zellij and should not be used in a production run, especially the <tt>-random</tt> method. @subsection autotoc_md8 Partial Parallel Output Mode There is a <em>partial parallel output</em> mode in which you can tell Zellij to only output a portion of the parallel decomposed files. This is selected with the <tt>-start_rank \<rank\></tt> and <tt>-rank_count \<count\></tt> options. In this case, Zellij will only output the ranks from <tt>rank</tt> up to <tt>rank+count-1</tt>. For example, if you run <tt>zellij -ranks 10 -start_rank 5 -rank_count 3</tt>, then zellij would output files for ranks 5, 6, and 7. This is somewhat inefficient since zellij will do many of the calculations for all ranks and only output the specified ranks; however, it does allow you to run multiple copies of zellij simultaneously. For example, you could run: @icode zellij -ranks 16 --start_rank 0 --rank_count 4 zellij -ranks 16 --start_rank 4 --rank_count 4 zellij -ranks 16 --start_rank 8 --rank_count 4 zellij -ranks 16 --start_rank 12 --rank_count 4 @endicode simultaneously and all 16 files should be output faster than running a single execution that wrote all of the files. @subsection autotoc_md9 Parallel Capable Parallel Execution If Zellij is compiled with parallel capability enabled (This is shown at the beginning of the <tt>-help</tt> output or the version information output when zellij begins executing as <tt>Parallel Capability Enabled</tt>), then you can run Zellij in parallel using the normal <tt>mpiexec -np \<\#\> zellij \<normal zellij options\></tt> command. In this case, there will be <tt>\#</tt> copies of zellij running simultaneously and each copy will divide up the output files and work among each process/copy. For example, if you run <tt>mpiexec -np 8 zellij -ranks 1024 -latice lattice.txt</tt>, then there will be 8 copies of zellij running and each will output <tt>1024/8 = 128</tt> output files. @subsection autotoc_md10 Maximum Open File Complications Most compute systems have a limit on the number of files that a program can have open simultaneously. For many systems, this limit is 1024. The files that zellij deals with are (1) the unit cell meshes and (2) the per-rank output files, and (3) the standard input, output, and error files. Because of this, it is somewhat easy for a zellij execution to exceed the open file limit. Zellij attempts to handle this automatically using logic similar to: * If the unit cell count exceeds the open file limit, then close each unit cell after each access before opening the next unit cell mesh. * If the number of <tt>-ranks</tt> that zellij is creating exceeds the open file count, then determine how many output files can be open at one time (max_open = open file limit - 3 - number of unit cells open simultaneously) and run zellij in a <tt>subcycle</tt> mode where it is only writing to <tt>max_open</tt> files at one time. * If the <tt>max_open</tt> calculated in the above bullet is too small, then set the mode to only open a single unit cell mesh at a time and redo the calculation. * If all else fails, run with only a single unit cell file open and only a single output mesh rank file open. If the above logic fails and Zellij is unable to run without exceeding the open file count, you can specify the behavior manually using a combination of the <tt>-minimize_open_files=\<UNIT|OUTPUT|ALL\></tt> option and the <tt>-subcycle</tt> and <tt>-rank_count \<\#\></tt> options. The options to <tt>-minimize_open_files</tt> are: * <tt>UNIT</tt> - only have a single unit cell mesh open at one time; close before accessing another unit cell mesh. * <tt>OUTPUT</tt> - only have a single output rank mesh file open at one time. * <tt>ALL</tt> - both of the above options. The <tt>-subcycle</tt> and <tt>-rank_count \<\#\></tt> options cause zellij to output <tt>\#</tt> output files at a time and then cycle to the next <tt>\#</tt> output files until all files have been output. For example, <tt>zellij -ranks 1024 -subcycle -rank_count 256</tt> would do the following: * First cycle would output ranks 0 to 255, * Second cycle would output ranks 256 to 511, * Third cycle would output ranks 512 to 767, * Fourth cycle would output ranks 768 to 1023. In this mode, there will the <tt>\#</tt> output files open simultaneously (unless <tt>-minimize_open_files=OUTPUT|ALL</tt> was specified also). So the total number of open files will be <tt>unit cell count + 3 + \#</tt> or <tt>1 + 3 + \#</tt> if <tt>-minimize_open_files=UNIT</tt> was specified. @section autotoc_md11 Execution Complexity Zellij is intended to produce extremely large meshes and is therefore very concerned with both memory efficiency and execution time efficiency. @subsection autotoc_md12 Memory Complexity Zellij stores the following data: * For each unit cell template mesh: * metadata * 64-bit Ids of nodes on each min_I, max_I, min_J, max_J face * For each entry in the lattice definition: * metadata (approximately 1KiByte) * temporarily it will hold 64-bit Ids of nodes on the max_I and max_J faces. This will be deleted once the upper <tt>I</tt> and upper <tt>J</tt> "neighbor" entry has been processed (see below) * For the lattice: * vector containing the lattice definition. The main memory use once the output file is being processed is the temporary storage containing the nodes on the <tt>max_I</tt> and <tt>max_J</tt> faces. The lattice is processed cell by cell. For an <tt>II by JJ</tt> sized grid, the cells are processed in the order <tt>(1,1), (2,1), ... , (II, 1), (1,2), (2,2), ..., (II, JJ)</tt>. The temporary storage on the <tt>max_I</tt> face is only needed until the next cell is processed. That is, for cell <tt>(i,j)</tt>, its <tt>max_I</tt> nodes will be used during the processing of cell <tt>(i+1, j)</tt> and then deleted. The temporary storage on the <tt>max_J</tt> face is retained for a longer time. For cell <tt>(i,j)</tt>, the <tt>max_J</tt> storage is needed for cell <tt>(i, j+1)</tt> and then deleted. For a grid of size <tt>(II, JJ)</tt>, there will at most be: * 1 temporary vector of size <tt>max_I</tt> nodes * <tt>II</tt> temporary vectors of size <tt>max_J</tt> nodes. If you have a lattice that is rectangular (<tt>II != JJ</tt>), then it is more efficient for memory usage to make the <tt>I</tt> direction the smallest value if possible. In addition to the above memory usage, zellij must also transfer the mesh coordinate data and element block connectivity data for each lattice entry to the output file. Zellij outputs the model using the following pseudo-code: @icode for each j : J for each i : I read cell(i,j) x, y, and z local coordinates map coordinates to offset in output mesh eliminate nodes that join to an already output neighbor cell write cell(i,j) x, y, and z global coordinates for each j : J for each i : I for each element block in cell(i,j) mesh read block connectivity map local node ids to global node ids write block connectivity @endicode The maximum memory use will be the size of storage needed for the <tt>x</tt> <tt>y</tt> and <tt>z</tt> coordinates of a unit cell mesh or the storage needed to hold the connectivity for a single unit cell element block. Note that the memory requirements are proportional to the size of an individual unit cell mesh and not a function of the size of the output mesh. It is possible to create meshes which are much larger than the amount of memory present on the compute system running zellij. The memory being used by zellij during execution will be output if the <tt>--debug 2</tt> argument is specified at execution time. @subsection autotoc_md13 Execution Time Complexity For a large model, the majority of the execution time is related to: * Read/process/write element block connectivity * Read/process/write nodal coordinates * Categorize boundary nodes on each unit cell mesh @subsection autotoc_md14 Efficiency at the NetCDF level The Exodus format which is used for the unit cell template meshes and the output mesh uses the NetCDF library for on-disk storage. There are several variants of the NetCDF on-disk storage including the format: <tt>netcdf3</tt>, <tt>netcdf4</tt>, and <tt>netcdf5</tt> and the integer size (32-bit integers or 64-bit integers). Although these details are usually transparent to the user, they can affect the execution time especially when very large meshes are being processed. @subsubsection autotoc_md15 Format The <tt>netcdf3</tt> format is the original native NetCDF format. At the time the library was being developed, the <tt>byte endianness</tt> of data stored on disk was not standard among the computes in use at that time and the NetCDF developers had to pick an <tt>endianness</tt> for the data. They picked the XDR standard which stood for <em>eXternal Data Representation</em> which was used for communicating between different computer systems. Regretfully, the representation used by XDR turned out to be opposite of the representation used by (almost?) all systems in use today, so each read and write of data in the <tt>netcdf3</tt> format results in a translation of the endianness. This translation is very fast, but is overhead that would not be needed if the on-disk format was the opposite representation. This representation is also used by the <tt>netcdf5</tt> format. However, the NetCDF <tt>netcdf4</tt> format is based on using the HDF5 library to manage the underlying data format on disk and it can read and write data using the native endianness of the system on which the data is being read and written and therefore does not incur the cost of transforming the data's endianness. @subsubsection autotoc_md16 Integer Size By default, most current mesh generators will output a mesh using 32-bit integer data. This is sufficient to represent a mesh with up to approximately 2.1 billion nodes and elements. If the input mesh and the output mesh have the same integer size, then there is no data conversion needed. The data will be read as <tt>N</tt>-bit integers, processed as <tt>N</tt>-bit integers, and written as <tt>N</tt>-bit integers. However, if the input mesh is <tt>N</tt>-bit integers and the output mesh is <tt>M</tt>-bit integers, then the NetCDF library will convert all integer data (element block connectivity typically) from <tt>N</tt> bits to <tt>M</tt> bits which for large meshes can incur an execution time overhead. @subsubsection autotoc_md17 Compression The NetCDF library supports compression of the output file. Typically, the <tt>zlib</tt> compression algorithm is used, but recently NetCDF begain supporting the <tt>szip</tt> compression and a few more algorithms are starting to be supported. The benefit of the compression is that it can result in much smaller output (and input) mesh files; the disadvantage is that the default <tt>zlib</tt> compression algorithm is not very fast and can increase the execution time of zellij. The <tt>szip</tt> compression algorithm is faster with typically (but not always) slightly less compression, but it still will incur an overhead in execution time. @subsubsection autotoc_md18 Recommendations For minimal overhead, it is recommended that: * Use the <tt>netcdf4</tt> format for all input and output meshes * Use the same integer size for all input and output meshes * The integer size of the output mesh can be specified using the <tt>-32</tt> or <tt>-64</tt> options. * The <tt>-64</tt> option is the default. It is most efficient if the format and integer size of the input mesh matches the output mesh. The format of the input meshes can be converted using the <tt>io_shell</tt> application with the <tt>-netcdf4</tt> and <tt>-64</tt> or <tt>-32</tt> options. The format and integer size of a mesh can be queried using the <tt>exo_format</tt> application. For illustration, here is the execution time for several runs with different format and integer size.
